# ADR 0001 — Athlete cards design

**Status:** Accepted (workshop, 2026-04-27)
**Owners:** Matthias
**Supersedes:** —
**Superseded by:** —

## Context

The Lifter Lookup tab today presents history as charts and a meet-by-meet table — accurate but un-shareable. The "athlete cards" feature (Arc 6 / LL3) ships a single-screen visual summary per lifter that a teammate can screenshot or download to share to Instagram / Discord / a coach.

Two things blocked implementation when we sized the work:

1. **Backend data layer** — does the OpenIPF parquet expose meet-level metadata (federation, sanctioning level, prestige) so the card can distinguish "IPF Worlds" from a small provincial open?
2. **Design open questions** — tier styling, image-export library, mobile breakpoint, stat-callout list, animation.

This ADR records the answers so the future implementation session (Session D) can ship without re-litigating.

## Backend data audit

Inspected `data/processed/openipf.parquet` directly via `DESCRIBE openipf` and grouped meet-level rows. Findings:

| Column | Type | Athlete-card relevance |
|---|---|---|
| `MeetName` | VARCHAR | Primary signal for tier inference. Free-text but stable enough that the top-50 names cover ~80% of meets. |
| `Federation` | VARCHAR | "CPU" for all in-scope rows (Canada+IPF). Not discriminating. |
| `ParentFederation` | VARCHAR | "IPF" for all in-scope rows. Not discriminating. |
| `MeetCountry` | VARCHAR | Useful for "international" tier flagging when non-Canada. |
| `Place` | VARCHAR | Direct podium signal — `'1'` / `'2'` / `'3'` are the medal positions. |
| `Date` | TIMESTAMP | Powers career arc / "longest streak" / "biggest jump" callouts. |
| `Goodlift` | DOUBLE | Per-meet IPF GL Points — powers "all-time PR by GLP" callout. |

**Verdict: no schema change needed.** Meet-level columns are present. Tier inference can run from `MeetName` substring patterns alone. Top-30 distinct `MeetName` values cover ~25k of the rows (out of ~50k total in scope), so a curated catalogue of ~50 patterns gets us most of the coverage with a sane long-tail fallback.

## Design decisions

### 1. Tier model — vendored catalogue, not derived column

Mirror the existing `PROVINCE_CATALOGUE` pattern at `frontend/src/tabs/QtLiveCoveragePanel.tsx:63-82` but for meet prestige. New file:

```
frontend/src/data/meet_prestige_catalogue.json
```

Schema:

```json
[
  {
    "pattern": "(?i)^(canadian )?nationals?$|^canadian championship",
    "tier": "national",
    "weight": 100
  },
  {
    "pattern": "(?i)^western canadian|^central canadian|^eastern canadian",
    "tier": "regional",
    "weight": 50
  },
  {
    "pattern": "(?i)provincials?$|provincial championship",
    "tier": "provincial",
    "weight": 30
  },
  { "pattern": "(?i)open$|open and|fall classic|winter open", "tier": "local", "weight": 10 }
]
```

A meet's tier = the highest-weight pattern that matches `MeetName`. A `MeetCountry !== 'Canada'` meet auto-elevates to `international`. Matches happen client-side at render time — no backend change, no preprocess change.

Why a JSON file (Vite static import) and not an inline array: matches the proven `frontend/src/data/backtest_results.json` pattern (`About.tsx:8`), can be regenerated by a future `data/build_meet_catalogue.py` script if we want to mine the parquet for top-N coverage.

### 2. Tier styling — flat color tokens, not metallic gradients

Locked palette per `~/.claude/projects/.../memory/project_color_tokens.md` (coral `#FB923C` is the only orange-family color sitewide; indigo / teal / coral / 6-color non-orange palette mapped to chart roles; no ad-hoc hex).

Add `frontend/src/lib/colors.ts` (new file — confirmed not present today):

```ts
export const TIER_TOKENS = {
  international: { ring: 'ring-amber-300/70', text: 'text-amber-200', bg: 'bg-amber-950/30', label: 'International' },
  national:      { ring: 'ring-amber-400/70', text: 'text-amber-300', bg: 'bg-amber-900/20', label: 'National'      },
  regional:      { ring: 'ring-zinc-300/60',  text: 'text-zinc-200',  bg: 'bg-zinc-800/40',  label: 'Regional'      },
  provincial:    { ring: 'ring-zinc-400/40',  text: 'text-zinc-300',  bg: 'bg-zinc-800/20',  label: 'Provincial'    },
  local:         { ring: 'ring-zinc-500/30',  text: 'text-zinc-400',  bg: 'bg-zinc-900/40',  label: 'Local'         },
} as const
```

Reasoning: metallic gradients (gold/silver/bronze CSS) look gimmicky against the dark zinc theme already locked in CLAUDE.md. The amber-for-prestige + zinc-for-everything-else pattern reuses the existing palette without introducing new hue families.

### 3. Stat callouts — three on the card, computed client-side

Card surfaces three callout chips above the chart:

| Callout | Definition | Compute |
|---|---|---|
| **All-time PR (Total)** | `max(TotalKg)` across all SBD meets | Client-side from existing `/api/lifter/history` response |
| **All-time PR (GLP)** | `max(Goodlift)` across all SBD meets | Client-side from existing `/api/lifter/history` response |
| **Highest tier reached** | Highest-weight tier in lifter's meet history (e.g. "International" if any non-Canada IPF meet) | Client-side, applies catalogue to each meet's `MeetName` |

All three derive from the existing `/api/lifter/history` payload — no new endpoint, no schema change.

Two stretch callouts (longest streak, biggest jump) are intentionally **out of scope for the first ship**. They require either gap-detection logic or year-over-year deltas that aren't in the API today. Add them after the v1 card lands and we have a use-pattern signal.

### 4. Image export — `html-to-image`, lazy-loaded

Survey:

| Library | Gzipped size | Notes |
|---|---|---|
| `html-to-image` | ~6 KB | Modern API (`toPng`, `toBlob`), TypeScript-native |
| `dom-to-image-more` | ~12 KB | Older API, fork of the original |
| `html2canvas` | ~46 KB | More robust on weird CSS but ~8x bigger |

**Pick: `html-to-image`.** A coral/zinc-themed flat card has no exotic CSS that html2canvas would need. Bundle delta is rounding error against the existing 357 KB Recharts chunk.

Wire it as a dynamic import inside a new `frontend/src/lib/exportCard.ts` helper so it ships on the same chunk as the AthleteCard component, not on the main bundle.

```ts
// exportCard.ts
export async function exportCardToPng(node: HTMLElement, filename: string) {
  const { toPng } = await import('html-to-image')
  const dataUrl = await toPng(node, { pixelRatio: 2, cacheBust: true })
  // trigger download
  const a = document.createElement('a')
  a.href = dataUrl
  a.download = filename
  a.click()
}
```

### 5. Layout — one breakpoint, no compact/expanded toggle

Card is a single fixed-aspect frame (3:4 portrait — matches Instagram-story-friendly proportions). On desktop and mobile alike it renders the same content; only the surrounding container width changes.

No "compact / expanded" toggle. The card *is* the compact view; the existing chart + meet table on `LifterDetail` *is* the expanded view. Two views, two purposes, no in-card mode-switch UI.

### 6. Animation — CSS only, no Framer Motion

No new animation dependency. The card fade-in on first paint uses the existing Tailwind v3 `transition-opacity` + `duration-500`. Tier-ring shimmer (if added later) is `animate-pulse` from Tailwind, not a JS lib.

Rationale: Framer Motion is 30-50 KB gzipped — not bundle-warranted for one card.

## Consequences

**Positive:**
- One PR ships the feature end-to-end. No backend coupling means CI changes nothing.
- Catalogue can grow over time without code churn — JSON edits, no rebuild logic.
- Bundle stays disciplined; html-to-image is light enough that the lazy-load discipline already in place absorbs it cleanly.
- Tier model has a fallback (`local`, weight 10) so any unmatched `MeetName` still renders a card.

**Negative:**
- The catalogue is hand-curated. Top-30 names cover ~80%; the long tail will render as `local`. We accept this as a v1 trade-off; a future `data/build_meet_catalogue.py` can crowd-source the tail.
- "International" tier elevation is only via `MeetCountry !== 'Canada'`. A Canadian lifting at IPF Worlds *held in Canada* would not get the international flag. Acceptable v1 simplification.

**Neutral:**
- No backend / preprocess / GHA changes.
- No new CI gates beyond what `npm run build` and the existing Vitest suite already do.
